# Education Data Governance Case: Pre-Submission Review

## Input Summary

审查高校教育数据治理能力与风险防控机制论文。

## Skill Invoked

`edtech-pre-submission-reviewer`

## References Used

- `review-rubric.md`
- `method-evidence-checklist.md`
- `ethics-data-governance.md`

## Main Output

## Overall Verdict

`Major revision before submission`。主题重要，但必须证明研究对象是“数据治理能力”，不是“数据平台建设”。

## Review Depth

`Strict`。该输出模拟投稿前审查，重点执行数据治理生命周期、高风险教育 AI、人类复核和申诉纠错检查。

## Findings

| Severity | Finding | Repair |
|---|---|---|
| CRITICAL | 若涉及学生画像、预警或管理决策，必须说明授权、人工复核和申诉机制 | 加入高风险数据使用治理表 |
| MAJOR | 数据治理能力维度需要理论来源和专家验证 | 建立文献编码 + Delphi 路线 |
| MAJOR | 风险防控机制不能只写原则 | 加入风险台账、审计追踪、删除流程和责任主体 |
| MAJOR | 平台截图不能证明治理效果 | 增加流程审计、访谈和质量指标 |
| MAJOR | 文献综述若称“系统梳理”需满足 PRISMA 风格透明度 | 补检索式、数据库、筛选流程和纳排理由，或改称“初步文献梳理” |

## V1.5 Gate Result

| Gate | Result |
|---|---|
| NIST-style lifecycle | Not yet pass: 缺少风险测量和持续监测 |
| Human oversight | Not yet pass: 需明确谁能否决或纠错系统输出 |
| PRISMA | Conditional: 仅在声称系统综述时适用 |
| COREQ | Conditional: 访谈可用，但需补报告细节 |

## 72-Hour Fix Plan

1. 加入数据最小化、用途限定、访问控制、保留删除和申诉纠错说明。
2. 用表格区分数据平台能力、数据治理能力和教育治理效果。
3. 明确哪些结论只能说“机制可行”，不能说“治理效能显著提升”。

## Remaining Gaps

需要核验中国高校教育数据治理制度样本和中文期刊文献。
