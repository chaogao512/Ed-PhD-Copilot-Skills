# Theory-to-Design Mapping

Use this table to prevent decorative theory.

| Theory concept | Design principle | System or governance element | Evidence indicator |
|---|---|---|---|
| Collaborative governance: shared decision-making | Create cross-role decision nodes | Joint dashboard review, governance meeting, escalation protocol | Decision participation, conflict resolution time, stakeholder perceived legitimacy |
| Socio-technical fit | Optimize organization and system together | Workflow redesign plus data platform | Workflow completion, adoption, workload, satisfaction |
| Human-in-the-loop | Keep humans responsible for high-stakes interpretation | Review, override, explanation and appeal interface | Override rate, explanation usefulness, error handling cases |
| PDCA | Continuous improvement cycle | Plan-do-check-act governance loop | Cycle completion, corrective actions, improvement records |
| Activity theory contradiction | Identify tensions among tools, rules and labor division | Contradiction map and redesign path | Observed contradictions, revised rules, actor interviews |
| Data governance | Clarify ownership, standards and quality | Metadata, authorization, quality checks | Data completeness, timeliness, error rate, access audit |

## Rule

Every major module in the paper must be traceable to at least one theory concept and one evidence indicator.
