# `Supervisor-Skills` 与教育信息化治理方向的不匹配点诊断框架

## 一、 核心评价体系的不匹配 (`Idea Evaluator` 模块)

原项目的想法评估器完全是建立在**“纯计算机技术性能”**的考量之上的，这与教育治理的“复杂社会系统”本质不匹配。

- **技术至上主义倾向**：原版过度关注算法是否达到 SOTA、参数量、计算成本等。在教育信息化治理中，一个技术即使性能再高，如果无法适配现有的学校科层管理体制，也是一个“坏 Idea”。
- **缺少“人”与“组织”的维度**：原版评估指标中，完全缺乏对**政策合规性（教育政策导向）**、**伦理风险（学生隐私/算法偏见）**以及**教师技术接受度（数字倦怠/培训成本）**的考量。
- **提示修改点**：接下来需要重构该模块的评估多维矩阵，引入教育管理与组织变革的考核因子。

## 二、 论文叙事与逻辑链条的不匹配 (`Introduction Drafter` & `Templates`)

原版的引言和论文模板遵循的是**“纯技术报告/算法改进”**的线性逻辑，无法承载交叉学科的论证厚度。

- **引言逻辑过于单薄**：原版的 Introduction 流向图是标准的“任务 \rightarrow 现有模型缺陷 \rightarrow 提出新网络/新损失函数”。而教育技术/治理论文的引言，核心在于**“现实管理体制瓶颈”与“技术赋能潜在可能”之间的张力**。
- **方法论（Methodology）的错位**：原版技术模板默认你的研究方法是“代码实现 + 跑分测试”。但你的专业属于交叉学科，往往需要**设计科学研究（DSR）**、**混合方法（定量系统效能 + 定性师生访谈/专家论证）**。原版模板在实验和论证部分完全缺失了社科实证的结构。
- **提示修改点**：接下来需要重构 Introduction 的推导流程，并将 Tech/Benchmark 模板改造为适合“智能校园架构/教育评价指标体系”的混合研究结构。

## 三、 视觉呈现与架构拓扑的不匹配 (`Figure Design Advisor` 模块)

原版的图表顾问默认你画的是“神经网络结构”或“纯数据流转”。

- **缺乏层级治理视角**：在教育信息化治理（如基于 Agent 的智能校园）中，图表不仅要展示技术的 API 调用，更要展示**教育行政、学校领导、教师、技术系统之间的多主体协同关系**，以及“自上而下”的治理层级。原版的画图建议无法指导你处理这种“技术-组织”的双层拓扑图。
- **提示修改点**：接下来需要为该模块注入“多层架构（基础设施-数据中台-Agent治理-业务应用）”和“多主体交互”的画图先验规则。

## 四、 学术话语体系与审稿机制的不匹配 (`Pre-Submission` & `Handbook`)

原项目完全是围绕“计算机顶会（ACL, KDD, SIGMOD 等）”的审稿文化搭建的，这与教育技术类的期刊/会议（如 SSCI/CSSCI 阵营）有巨大鸿沟。

- **文风“AI腔/技术腔”过重**：原版的润色和自查倾向于使用大量硬核的技术词汇。如果直接用于教育学领域的审稿，容易被评审专家批为“缺乏教育学理论底蕴”、“技术堆砌”。
- **缺失社科审稿的硬性红线**：原版的 Checklist 检查的是代码开源、Baseline 对比。它完全漏掉了教育研究中至关重要的**伦理审查（知情同意、数据脱敏）和政策依据自查****和**。
- **提示修改点**：接下来需要过滤掉模块中的“纯技术黑话”，补充教育学术语库，并增加社科类论文的合规 Checklist。
